# PRD（1枚）：〆トラ（MVP）

## 0. 概要

- プロダクト名：〆トラ
- 1行説明（誰の何をどう良くする）：就活中の大学生が、ES/説明会/面接の締切を登録し、締切前に通知で“出し忘れ”を防ぐ

## 1. 解決したい課題（Problem）

- ターゲット（Who）：就活を開始〜選考中の大学生（応募社数が増えて締切管理が破綻し始めた層）
- 困りごと（Pain）：締切が分散し見落としが起きる。締切ミスの機会損失/ストレスが大きい
- 今の代替手段（Alternatives）：Notion / Googleカレンダー / ToDo / スプレッドシート（入力・更新が面倒、就活の状態管理が弱く継続しづらい）
- なぜ今やるのか（Why now）：就活は短期集中で締切密度が高く、今すぐ価値が出る

## 2. 提供する解決策（Solution）

- 価値提案（Value Proposition）：締切管理の操作コストを最小化し、通知の信頼性で締切ミスを減らす
- 差別化ポイント（Why us）：汎用ToDoではなく「締切×ステータス×リンク」に縦特化したUI + 複数タイミングの通知
- コア体験（1つに絞る）：締切を登録 → 締切が近い順に確認 → 締切前に確実に通知

## 3. MVPスコープ（必須）

### 3.1 ユーザーフロー（3〜7ステップ）

1. LP/SNSからアクセス
2. メールでサインアップ（メール確認完了をもってSignupと定義）
3. 締切アイテムを2件登録（企業名・締切日時・種別・リンク）
4. ダッシュボードで締切近い順に確認し、ステータス更新
5. 締切前にメール通知を受け取る
6. 無料枠到達 or 複数通知が欲しくなったらProに課金

### 3.2 MVP機能（Must）

- [ ] Must-1：ログイン（メール）
- [ ] Must-2：締切アイテムCRUD（企業名/種別/締切日時/ステータス/リンク/メモ）
- [ ] Must-3：ダッシュボード（締切が近い順、ステータス即更新）
- [ ] Must-4：メール通知（Free=24h前、Pro=72h/24h/3h）
- [ ] Must-5：無料枠制限（10件まで）+ ペイウォール
- [ ] Must-6：Stripeサブスク（Pro 980円/月）+ 課金状態で機能切替

### 3.3 やらないこと（Won’t / Not now）

- [ ] Won’t-1：求人・締切の自動収集（スクレイピング等）
- [ ] Won’t-2：LINE通知 / Push通知（MVPではメールのみ）
- [ ] Won’t-3：ES自動生成/添削
- [ ] Won’t-4：コミュニティ/SNS機能
- [ ] Won’t-5：企業別の分析ダッシュボード（高度な可視化）
- [ ] Won’t-6：大学/学年などの個人属性の深掘り収集（PII最小）

## 4. マネタイズ（収益化）

- 課金モデル（例：サブスク/買い切り/広告/手数料）：サブスク
- 価格（仮）：Pro 980円/月
- 課金ポイント（どのタイミングで何を有料にするか）：
  - Free：アイテム10件まで、通知は24h前のみ
  - Pro：アイテム無制限、通知は72h/24h/3h、（将来）テンプレ/エクスポート
- ユーザーが払う理由（支払動機）：締切ミスの機会損失を避けたい。応募が増えるほど管理コストが上がり、複数通知と無制限が必要になる

## 5. 成功指標（最初の2週間）

- Signup数：`signup_users`
  - Signup定義（固定）：メール確認が完了し、ユーザー作成が確定した時点
- 計測イベント仕様（ソースオブトゥルース）：`docs/ANALYTICS_SPEC.md`
- アクティベーション（例：初回で価値体験した割合）：
  - Activation定義：サインアップ後24時間以内に「締切アイテムを2件作成」したユーザー
  - KPI：Activation率 = `activated_users / signup_users`（目標：35%）
- 継続（例：7日後残存率）：
  - D7継続率 = `d7_active_users / d0_signup_users`（目標：15%）
  - D7 Active定義：7日目±1日にダッシュボードを1回以上閲覧
- 収益（例：課金率、ARPU）：
  - 2週課金率 = `paid_users / signup_users`（目標：3%）
  - 2週MRR先行指標 = `paid_users * 980円`（目標：10,000円以上＝10人課金）
- 北極星指標（North Star Metric）：
  - 期限内に処理できた締切数（代理指標）
    - 計測定義（固定）：締切時刻（`deadline_at`）より前に、ステータスが「提出済」へ更新された件数
    - 計測方法（固定）：DBから `status_changed_at < deadline_at` を満たす件数を集計（イベントに本文は載せない）

## 6. 仮説と検証計画（最速で）

- 最大のリスク仮説（これが外れたら終了）：SNS→登録まで集客でき、かつ登録者が2件入力して通知価値を感じる
- 48時間でできる検証：
  - SNSショート動画を最低6本投稿（2フォーマット×3本）→ LPへ誘導
  - LPからFreeサインアップへ誘導（waitlistは作らない）
  - 24h以内に締切2件登録までの到達率を確認
- 成功/失敗の判定基準（Kill criteria）：
  - KILL-A（集客失敗）：2週間で `signup_users < 30`
  - KILL-B（価値失敗：signup_users >= 60 のとき判定）：
    - Activation率 `< 20%` または D7継続率 `< 8%`
  - KILL-C（マネタイズ失敗：activated_users >= 20 のとき判定）：`paid_users = 0`
  - 補足（誤判定防止）：`30 <= signup_users < 60` の場合は、投稿本数/導線を増やして追加1週間検証してからKILL判定する
  - 上記のうち2つ以上該当：本MVPは停止し、訴求/ターゲット/別案へピボット

## 7. 制約・前提

- 期限：2週間でMVP公開
- 予算（月額/初期）：月額 0〜10,000円（初期は極小、必要なら段階的に増やす）
- 開発体制：1人開発
- 法的/規約/年齢制限など：
  - PIIは最小（メールのみ）
  - 課金はStripeに委譲
  - 求人情報の自動収集はしない（規約リスク低減）
